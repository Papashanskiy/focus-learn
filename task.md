# Задача

Разработай локальное Python-приложение для ежедневной подготовки middle+/senior Python-разработчика к собеседованиям.

Рабочая директория: текущая директория проекта.
Используй только относительные пути.
Не используй `/home/user/`.

## Цель

Приложение должно помогать мне каждый день заниматься 60 минут:
1. проходить тему;
2. отвечать на вопросы;
3. получать подсказки;
4. сверять ответ с эталонным;
5. получать AI feedback;
6. видеть прогресс.

Я Python middle+ разработчик и хочу дорасти до senior. Приложение делается только для меня.

## Формат приложения

Первая версия: локальное терминальное приложение.

Предпочтительный вариант: TUI-интерфейс в терминале, например на Textual/Rich.
Обычные CLI-команды можно добавить для служебных операций.

Веб-интерфейс пока не делать, но архитектуру спроектировать так, чтобы его можно было добавить позже без переписывания ядра.

## Технические требования

- Python: использовать версию, доступную в окружении; если возможно, ориентироваться на Python 3.14.
- Хранение данных: SQLite.
- LLM: локальная Ollama.
- Предпочтительная модель: `qwen3-coder:30b`.
- Запуск: WSL2.
- Приложение должно работать локально.
- Уведомления не нужны.
- Экспорт результатов не нужен.
- Форматирование кода не нужно.

## Основные функции

### Сессии

- Команда/экран запуска ежедневной сессии.
- Сессия рассчитана на 60 минут.
- Пользователь может двигаться медленно, поэтому приложение не должно жестко требовать фиксированное число вопросов.
- История каждой сессии должна сохраняться.

### Темы

- Приложение должно иметь базу тем для Python backend/system design подготовки.
- Пользователь может выбрать тему.
- Приложение может предложить тему на основе прогресса.

### Вопросы

- В базе должны быть стартовые вопросы, сгенерированные через LLM.
- Вопросы должны быть разбиты по темам и уровню сложности.
- Должен быть режим “только вопросы”.
- Пользователь должен иметь возможность добавить свои вопросы через свободный текст: я пишу LLM, что хочу добавить, а приложение превращает это в структурированный вопрос и сохраняет в SQLite.

### Ответы и самопроверка

- Пользователь отвечает текстом.
- Приложение сохраняет ответ.
- Приложение показывает эталонный ответ.
- Пользователь выставляет самооценку.
- AI может дать краткий feedback: что хорошо, что упущено, что повторить.

### Прогресс и статистика

- Показывать статистику:
  - сколько сессий пройдено;
  - сколько вопросов отвечено;
  - средняя самооценка;
  - слабые темы;
  - последние сессии;
  - динамика по темам.

## Архитектура

Раздели проект на слои:

- domain: сущности и бизнес-правила;
- services: сценарии приложения;
- infra: SQLite, Ollama client, repositories;
- ui: TUI/CLI;
- tests: тесты.

Не смешивай UI, работу с БД и бизнес-логику.

## Документация и память проекта

Создай:

- `README.md` — как установить, запустить и пользоваться;
- `CLAUDE.md` — описание проекта, архитектуры и правил работы агента;
- `DEVELOPMENT_LOG.md` — дневник изменений;
- `ROADMAP.md` — план работ и текущий статус.

После каждой реализованной крупной части обновляй `DEVELOPMENT_LOG.md` и `ROADMAP.md`.

## Процесс работы

1. Сначала изучи текущую директорию.
2. Если есть блокирующие вопросы, задай максимум 5 вопросов.
3. Если вопрос не блокирующий, сделай разумное предположение и зафиксируй его в плане.
4. Составь план реализации.
5. Реализуй MVP.
6. Добавь тесты.
7. Запусти тесты.
8. Если Ollama недоступна, сделай fake/mock LLM client для тестов, а live Ollama-тесты пометь как optional.
9. Заверши работу только после проверки запуска приложения и тестов.

## Критерии готовности MVP

- Приложение запускается локально.
- Можно создать/начать сессию.
- Можно выбрать тему.
- Можно отвечать на вопросы.
- Ответы сохраняются в SQLite.
- История сессий сохраняется.
- Работает статистика.
- Есть интеграция с Ollama или fallback/mock, если Ollama недоступна.
- Есть тесты ключевой логики.
- Документация обновлена.

Важно: после составления плана не останавливайся. Сразу переходи к реализации MVP. Не завершай ответ только архитектурой или рекомендациями.

В финальном ответе укажи:
- что реализовано;
- какие команды запускались;
- какие тесты прошли;
- как запустить приложение;
- какие ограничения остались.
